## Wave 4 — Correção dos problemas F1–F6 do agente WhatsApp

Objetivo: eliminar perda de contexto, loops e alucinações identificados no diagnóstico, com verificação após cada passo (build verde + testes Deno verdes + 1 simulação ponta-a-ponta da fixture correspondente).

---

### Princípio: persistir estado, parar de re-derivar

Hoje cada turno re-deriva o passo do funil a partir do histórico bruto. Vamos adicionar um **estado persistente do funil por lead** + **fonte do nome no contato**, e usar isso como verdade nos overrides.

---

### Passo 1 — Migration: `lead_funnel_state` + `contacts.name_source` (resolve raiz de F1, F3, F4)

**Schema novo:**
```sql
-- Tabela de estado do funil (1:1 com leads)
CREATE TABLE public.lead_funnel_state (
  lead_id uuid PRIMARY KEY REFERENCES public.leads(id) ON DELETE CASCADE,
  step text NOT NULL DEFAULT 'abertura',
    -- abertura → nome → email → interesse → localizacao → levantamento → livre
  name_confirmed boolean NOT NULL DEFAULT false,
  email_confirmed boolean NOT NULL DEFAULT false,
  interest_confirmed text,
  location_known text,           -- 'spain' | 'outside' | null
  entry_date_confirmed text,     -- ISO date ou 'never'
  empadronado_confirmed boolean,
  outside_spain_progress jsonb DEFAULT '{}'::jsonb, -- {age, europa6m, familiar, remoto, formacao}
  last_step_change timestamptz NOT NULL DEFAULT now(),
  updated_at timestamptz NOT NULL DEFAULT now()
);

-- Coluna name_source em contacts
ALTER TABLE public.contacts
  ADD COLUMN name_source text NOT NULL DEFAULT 'AUTO';
  -- AUTO = WhatsApp ProfileName ou WhatsApp <phone>
  -- USER_CONFIRMED = digitado pelo cliente em resposta à pergunta de nome
  -- STAFF_EDITED   = editado manualmente por funcionário
```

**RLS** (segue padrão existente):
- `lead_funnel_state`: SELECT/INSERT/UPDATE/DELETE para staff (`ADMIN/MANAGER/SUPERVISOR/ATENCAO_CLIENTE/ATENDENTE_WHATSAPP`) + service_role (edge function); leitura adicional para os mesmos sectors da tabela `leads`.
- `contacts.name_source`: já coberto pelas RLS existentes da tabela.

**Backfill** (no mesmo migration):
- `INSERT INTO lead_funnel_state (lead_id, step) SELECT id, 'livre' FROM leads;` — leads existentes não vão tentar re-perguntar nada porque entram em `livre`.
- `UPDATE contacts SET name_source = 'STAFF_EDITED' WHERE full_name IS NOT NULL AND full_name NOT LIKE 'WhatsApp %';` — assume confiável o que já existe (não vamos re-perguntar nome a quem já tem cadastro).

**Verificação:** migration roda sem erro; `SELECT count(*) FROM lead_funnel_state` = `SELECT count(*) FROM leads`.

### Passo 2 — Helper `lib/funnel-state.ts` + integração com webhook (resolve F1, F4)

Criar `supabase/functions/whatsapp-webhook/lib/funnel-state.ts`:
- `loadFunnelState(supabase, leadId)` → carrega ou inicializa.
- `computeNextStep(state, contact)` → decide o próximo `step` baseado no que já está confirmado.
- `applyTurnUpdates(supabase, leadId, patch)` → grava mudanças de flags + step + `last_step_change`.
- `buildStateDirective(state, language)` → retorna 1-2 linhas em PT/ES/EN/FR para injetar no system prompt: `"ETAPA ATUAL: levantamento. NÃO peça novamente: nome, email, interesse, localização, data de entrada."`

**Integração em `index.ts`** (handler):
1. Após resolver `lead.id`, chamar `loadFunnelState`.
2. Concatenar `buildStateDirective` ao system prompt **antes** da chamada Gemini.
3. Após overrides finais, chamar `applyTurnUpdates` com flags derivadas do turno (e.g., se `extracted.name` veio + `contact.name_source` virou `USER_CONFIRMED`, marcar `name_confirmed=true` e avançar step para `email`).
4. Se `state.name_confirmed === true`, sempre passar `nameAlreadyKnown=true` em `forceSkipFullNameIfAlreadyKnown` (independente do que `isAutoGeneratedContactName` disser).

**Verificação:** rodar fixture do Gustavo (efa19624) → garantir que após confirmar nome `gustavo braga`, mesmo na 11ª mensagem o `state.name_confirmed=true` impede a re-pergunta.

### Riscos e mitigação

- **Backfill de leads antigos como `livre`**: nunca re-perguntará dados, mas também não validará. Aceitável — não regride comportamento atual.
- **`name_source` default `AUTO` para contatos pré-existentes**: o backfill marca como `STAFF_EDITED` para evitar re-pergunta indevida. Se algum nome legítimo veio do ProfileName e for considerado verdadeiro, fica como `STAFF_EDITED` — sem perda.
- **Mudança no system prompt** (diretiva de etapa): pode alterar tom da resposta da Gemini. Mitigação: diretiva curta e imperativa, posicionada após o prompt-base existente, sem alterar persona.
